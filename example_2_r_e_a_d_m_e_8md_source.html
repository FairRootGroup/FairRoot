3 ##Tutorial1:
4 How to implement a simple detector using the old ASCII interface to describe the geometry (see Detector Geometry and Media for more details about this geometry interface)
5 See also Adding new detector for a detailed description of the detector class implementation, MC point class and the CMakeLists.txt
6 
7 To run this tutorial:
8 
9 After building FairRoot successfully go to the build directory and call the configuration script (config.sh or config.csh according to our shell)
10 cd to the Tutorial1/macros directory
11 run the following command: root run_tutorial1.C (you can also load the macro into root and then run it)
12 This will start a simulation session with Geant3 and create 10 events generated by a box generator, the input in the box generator can be chosen by the user, by default in this macro we use "pions" (see macro lines 20-22) for details.
13 To choose Geant4 you just have to change TGeant3 to TGeant4 in line 8 of the macro, or of you are loading the macro to root before exciting it you just give TGeant4 as second argument to the run_tutorial1 call, i.e:
14 
15 ```bash
16 root>.L run_tutorial1.C
17 root>run_tutorial(10, "TGeant4")
18 ```

28 ##Tutorial3
29 This example demonstrate
30 
31 detector simulation
32 event by event digitization
33 time based digitization
34 Message Queue based reconstruction: The Message Queue based reconstruction demonstrates the usage of FairMQ.
35 Any number of "sampler" processes (FairMQSampler, FairMQSamplerTask, TestDetectorDigiLoader) send the data generated in the digitization step to a number of "processor" processes which do the reconstruction with the data (FairMQProcessor, FairMQProcessorTask, FairTestDetectorMQRecoTask). After the reconstruction task, the processes send the output data to a number of "sink" processes, which write the data to disk (FairMQFileSink).
36 All the communication between these processes is done via FairMQ and can be configured to run over network (tcp) or on a local machine with inter-process communication (ipc).
37 Currently, the number of processes and their parameters are configured with bash scripts/command line parameters. The scripts are located in the macro directory, together with README.md for more details on how to configure them.

54 ##Tutorial6

56 #### Initialisation of parameter within MQ Tasks ####
57 
58 * Introduction
59 
60 This example demonstrates the usage of the runtimeDb services ( including comunication to SQL databases via FairDB )
61 in order to initialise parameter at a certain time within FairMQ Tasks.
62 Once a parameter has been initialised, it is cached together with its interval of validity.
63 Further calls to an update will only trigger an initialistation if the update time falls outside of the parameter
64 interval of validity.
65 In all other cases, a pointer to the valid parameter will be forwarded to the MQ process.

97 * Launch the FairDbMQ dispatcher
98 
99 With the FairMQDB the FairMQ Tasks do not directly communicate to the backend parameter store but
100 initialisation messages are routed through a continuously running process i.e the FairDbMQ dispatcher.
101 The user should launch the daemon once i.e
102 
103 ```bash
104 <my_fairroot_path>/build/bin/db_dispatcher &
105 ```
106 
107 The FairDbMQ dispatcher is then continuously listening to connection comming from
108 
109  * FairMQ Tasks requiring an initialisation
110  * Assigned FairDBMQWorker process to handle this request
111 
112 Once the FairDbMQDispatcher process is run, you should see the following prompt
113 
114 ```bash
115 -I- FairDbMQDispatcher:: Queueing Daemon Started ...
116 ```
117 meaning that process initialised itself properly and is running on the background.
118 
119 
120 
121 * Launch the Assigned FairDbMQWorker
122 
123 In order to handle an initialisation request, the user should launch at least one FairDbMQ worker
124 process which as the FairDbMQDispatcher is daemonized.
125 
126 ```bash
127 <my_fairroot_path>/build/bin/db_tut6worker &
128 ```
129 
130 Once the FairDbMQworker process is run, you should see the following prompt
131 
132 ```bash

136

151 * SQL Database settings
152 
153 Furthermore, in this example, the FairDbMQWorker will use an SQL server to fetch the updated parameters values if needed.
154 In FairDB services are used within the assigned worker, the user, before launching it, needs to define the connection to a Database system (MySQL, PostGreSQL or SQlite )
155 as it required by the FairDb Library i.e
156 
157 As an example, such a definition for a local MySQL server is done exporting the following environment varaiables:
158 
159 ```bash
160 export FAIRDB_TSQL_URL="mysql://localhost/r3b"
161 export FAIRDB_TSQL_USER="scott"
162 export FAIRDB_TSQL_PSWD="tiger"
163 ```
164 
165 A more complete script is provided for setting up the main supported RDBMS backends namely MySQL, PostGreSQL and SQLite:
166 
167 ```bash
168 <my_fairroot_path>/build/bin/dbconfig.sh
169 ```
